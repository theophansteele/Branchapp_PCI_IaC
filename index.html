Out of Scope (no CHD, no CDE network access)

PCI DSS v4.0 Controls Applied to In-Scope Systems

Requirement 1
Install and Maintain Network Security Controls
  • AWS Security Groups: default-deny, explicit 443 allow
  • GCP VPC Firewall: deny-all ingress/egress baseline
  • Cloudflare WAF in front of all CDE endpoints
  • Network segmentation between CDE and non-CDE
  • AWS Config rule: restricted-ssh, restricted-common-ports
EC2GCPCloudflareK8s
Requirement 2
Apply Secure Configurations to All Components
  • EC2 IMDSv2 enforcement (Config rule)
  • GCP OS Login required (Org Policy)
  • Disable service account key creation (GCP)
  • No public IPs on GCP VMs (Org Policy)
  • Cloudflare: hide server version headers
EC2GCPCloudflare
Requirement 3
Protect Stored Account Data
  • KMS CMK with automatic annual rotation (AWS)
  • EBS encryption by default across all volumes
  • S3 SSE-KMS enforced on all CDE buckets
  • GCP CMEK on GCS with 90-day key rotation
  • RDS storage encryption (Config rule)
  • Drata monitors encryption state continuously
EC2/EBSS3RDSGCS
Requirement 4
Protect Cardholder Data with Strong Cryptography in Transit
  • S3 bucket policy: deny all HTTP (TLS-only)
  • Cloudflare: TLS 1.2 minimum, TLS 1.3 preferred
  • HSTS with preload (max-age 1 year)
  • 301 redirect all HTTP → HTTPS (Cloudflare)
  • AWS Config: s3-bucket-ssl-requests-only
S3CloudFrontCloudflarePub/Sub
Requirement 5
Protect All Systems Against Malware
  • GuardDuty: EBS malware scanning on EC2
  • GuardDuty: runtime threat detection
  • Drata monitors GuardDuty status continuously
  • Security Hub aggregates findings across all services
EC2EBSGuardDuty
Requirement 6
Develop and Maintain Secure Systems and Software
  • Security Hub PCI DSS standard enabled
  • Cloudflare OWASP Core Ruleset (Managed Rules)
  • Custom WAF: SQLi blocking, admin path protection
  • tfsec + Checkov on every pull request (GitHub Actions)
  • 19 AWS Config rules monitoring patch & config state
  • Drata: Jira integration tracks vulnerability remediation
EC2CloudflareGitHubJira
Requirement 7
Restrict Access to System Components by Business Need
  • S3 public access block on all CDE buckets
  • AWS Config: no public S3 read/write
  • IAM root access key check
  • GCP: restrict public IPs on VMs (Org Policy)
  • Cloudflare: block admin paths from non-internal IPs
  • Drata UAR across AWS, GCP, Cloudflare, Google Workspace
S3IAMGCP IAMCloudflare
Requirement 8
Identify Users and Authenticate Access to System Components
  • IAM Password Policy: 12+ chars, 90-day rotation, no reuse ×12
  • Config rule: MFA on all IAM console users
  • Config rule: root account MFA enabled
  • CloudWatch alarm: console auth failures
  • Cloudflare: rate-limit login endpoint (5 req/min)
  • Drata: continuous MFA compliance monitoring
IAMGCP IAMCloudflare
Requirement 9
Restrict Physical Access to Cardholder Data
  • AWS data center physical controls (AWS responsibility)
  • GCP data center physical controls (Google responsibility)
  • Drata: maps AWS SOC2/PCI reports as evidence
  • Document physical access policy in Confluence (Drata)

  • GCP Storage bucket locked retention policy
  • SNS alerts for all suspicious API activity patterns

Terraform Project Structure

Four modules map directly to PCI-scoped systems. Run terraform apply once to provision all controls; GitHub Actions re-plans on every push and runs a daily compliance health check via the AWS + Drata APIs.
terraform-pci-compliance/
.github/workflows/

modules/
aws-pci/
main.tf — KMS, CloudTrail, GuardDuty, Security Hub, Config ×19, SGs, EBS enc, CW alarms
gcp-pci/
main.tf — CMEK, Audit Log sink, VPC firewall, Org Policies, SCC
cloudflare-pci/
main.tf — TLS 1.2+, HSTS, OWASP WAF, Managed Rules, custom rules, HTTP→HTTPS
drata-setup/
main.tf — AWS cross-account IAM role + GCP service account for Drata read-only access

variables.tf — all inputs (AWS, GCP, Cloudflare, Drata)
outputs.tf — role ARNs to paste into Drata UI
providers.tf — AWS, Google, Cloudflare + S3 remote state backend
README.md — quickstart, secrets list, full control coverage table
05

Compliance as Code + Drata Workflow

1
terraform apply
Provision all PCI controls: security groups, KMS, CloudTrail, GuardDuty, WAF, Drata IAM roles. One command, all four providers.
2
Connect Drata
Copy terraform output role ARNs and service account emails into Drata Connections UI. 13 integrations enabled.
3
Drata Auto-Maps
Drata reads your AWS Config, GuardDuty, and GCP state and automatically maps each finding to the relevant PCI DSS v4.0 control.
4
Daily Monitor
GitHub Actions runs at 06:00 UTC — checks Config non-compliance, GuardDuty HIGH findings, Security Hub failed controls. Alerts Slack on failure.
5
PR Gate
Every Terraform PR runs tfsec + Checkov. Non-compliant infrastructure changes are blocked before merge. Plan posted to PR as comment.
6
Audit Ready
Drata continuously collects evidence. At audit time export the PCI DSS report — all 12 requirements mapped with automated evidence.
